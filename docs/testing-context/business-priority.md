# Business Priority

## 何が壊れると痛いか

| 領域 | 壊れた場合の影響 | 深刻度 |
|------|----------------|--------|
| 状態遷移の整合性 | タスクが不正な状態に入り、ワークフローが停止する | High |
| 承認の権限チェック | 権限のないユーザーが承認でき、レビュープロセスが形骸化する | High |
| タスク一覧の表示 | 全ユーザーが業務状況を把握できなくなる | Medium |
| 却下理由の保存 | 差し戻し理由が消え、担当者が対応方針を判断できない | Medium |
| タスク作成のバリデーション | 不正なタスクが永続化される | Low |
| UI のボタン表示制御 | 操作はできないがユーザーが混乱する | Low |

## Low / Medium / High の判断基準

- **High**: ユーザーが気づかないまま不正な状態が進行する。または業務プロセスの前提が崩れる
- **Medium**: ユーザーが気づける形で不具合が顕在化するが、業務に影響がある
- **Low**: ユーザー体験に影響するが、業務データの整合性には影響しない

## 運用影響

- 承認フローが壊れると、タスクが `pending_approval` で滞留し、チーム全体の進行が止まる
- 状態遷移バグは DB 上のデータ不整合を生むため、手動修正が必要になる
- UI の表示バグは問い合わせ増加につながるが、API 側が正しければデータは守られる
